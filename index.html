FCN介绍

参考文章：

全卷积神经网络(FCN)详解 - 知乎 (zhihu.com)

对于一个各层参数结构都设计好的CNN来说,往往要求固定输入的图片大小,比如AlexNet，VGGNet, GoogleNet等.

而FCN的好处就在于其可以让一个已经设计好的网络输入任意大小的图片.


1.CNN 和 FCN网络结构对比

(1)CNN

加入我们现在要设计一个区分猫狗和背景的网络,如果使用CNN的话架构应该如下:

假如输入的图像是14×14×3的彩色图像,则首先会经过5×5的卷积层,卷积层的输出通道数为16,于是可以得到10×10×16的一组特征图,接下来经过2×2的池化层,得到5×5×16的特征图,进行flatten展平操作后进入到两个50个神经元的全连接层,最后输出分类结果.

因为全连接层中输入层神经元数量固定,因此反推后卷积层的输入大小也要固定,这就不利于对不同尺寸的图片进行训练.


(2)FCN

全卷积神经网络,也就是说该网络中全是卷积层连接,如下图:

前面的卷积,池化步骤与之前的CNN是相同的,但是FNN在flatten的时候,FCN将其换成了一个卷积核大小为5×5,输出通道为50的卷积层,之后的全连接层都换成了1×1的卷积层(1×1的卷积其实相当于全连接操作)


为什么1×1的卷积相当于全连接?

CNN中神奇的1x1卷积 (qq.com)

(12条消息) 1×1卷积_霜桥月馆的博客-CSDN博客_1×1卷积

可以看到,FCN与CNN的区别在于FCN将CNN中的全连接层换成了卷积操作.换成全卷积操作后,由于没有了全连接层神经元数的限制,因此卷积层的输入可以接收不同尺寸的图像.

接下来的问题是,既然输入尺寸不一样,那么输出尺寸也不一样,如何理解?


2.FCN如何理解网络的输出?

(1)特征图尺寸变化

现在我们不考虑通道数,仅考虑上面网络中的特征图尺寸的变化,如下图(绿色为卷积核,橙色为特征图)

可以看到,输入是一张14×14大小的图片,经过5×5的卷积后(不做padding操作),得到10×10的特征图,然后进行2×2的池化后,尺寸缩小一半变为5×5的特征图,再经过5×5的卷积,特征图变为1×1,接下来就是进行两次1×1的卷积(也就是类似全连接操作),最终会得到一个1×1的输出结果.

这个1×1的输出结果,就代表最前面14×14的图像区域的分类情况.如果对应到最开始的猫狗和背景的分类任务,则最后输出的结果应该是1×3的矩阵,其中每个值表示14×14的输入图片中对应类别的分类得分.

在之前的讨论中,我们没有讨论通道情况,如果将网络放到正常包含通道的28×28×3的图像上,考虑特征图的通道数,此时输出值的对应情况如下:

上图中绿色区域表现了依次通过网络后的特征图尺寸变化情况。跟2(2)的图类似，因为这是一个猫狗和背景的三分类任务，所以最后输出的图像大小为8x8x3，以输出图像左上角绿色点为例，该点深度为3，对应输入图像的绿色区域，该点的3个值反应了输入图的绿色区域是分类为猫狗还是背景的得分情况。(可以看到,最后的1×1卷积操作可以很好地控制通道数,简化运算)

总的来说，FCN利用了输出结果和输入图像的对应关系，直接给出了输入图像相应区域的分类情况，取消了传统目标检测中的滑动窗口选取候选框。


4.FCN的优缺点

由于输出结果的每个值映射到输入图像上的感受野的窗口是固定的，也就是检测窗口是固定的，导致检测效果没那么好，但是速度却得到了很大的提升，而且可以输入任意尺寸的图片，为目标检测提供了一种新思路。

四、LSTM（长短期记忆）

参考链接：

人人都能看懂的LSTM - 知乎 (zhihu.com)

Illustrated Guide to LSTM’s and GRU’s: A step by step explanation | by Michael Phi | Towards Data Science

在普通的RNN当中，越晚的输入影响越大，而越早地输入影响则越小，并且无法更改这个逻辑。

而相对的，LSTM改变了这个逻辑，使用一套灵活的方案——只保留重要的信息。

首先来看一下LSTM的结构：

相比RNN而言，LSTM采用了门控装置，可以有选择性的存储信息。

我们来仔细看一下LSTM的内部结构：

（1）核心概念

LSTM的核心概念是cell state，以及不同的门控设施。

  • cell state可以理解为是网络的“记忆”，会沿着序列链“一路传递消息”

    • 可以在序列处理的过程当中不断地传递相关的信息。
    • 在LSTM当中，哪怕是最早的一些时间的信息都可以对后面的处理造成影响，这就解决了传统RNN的短期记忆问题
    • 在传递过程当中，信息可能会被添加或者移除（通过门控系统，也就是gate）

  • 门控系统是不同的神经网络，作用是决定哪些信息可以呈现在cell state当中。门控系统可以学习哪些知识是相关知识，从而学习如何保持或遗忘。


（2）Sigmoid

门控系统以sigmoid作为激活函数。sigmoid函数类似于tanh，但其会限制输出范围为0到1。

这样做的好处在于，可以很好地更新或者遗忘数据，这是因为：

  • 任何数乘0的结果都是0，通过这种方式就可以“遗忘”数据。
  • 任何数乘1的结果都是这个数，通过这种方式可以“保留”数据。

神经网络可以学习哪些数据是需要遗忘的，哪些则需要留下来。

LSTM当中一共有三种gate来构成门控系统，分别是：

  • forget gate
  • input gate
  • output gate


（3）forget gate

这扇门决定了哪些信息应该丢弃，哪些信息应该保留。来自前一个隐藏状态(hidden state)的信息和来自当前输入的信息通过sigmoid函数传递。值在0和1之间。越接近0表示忘记，越接近1表示保留。


（4）input gate

以下图为例，进行说明：

  • ①将hidden state和当前输入通过sigmoid激活函数。这一点和之前是类似的，输出的结果将在0到1之间，决定数据是否应该留下或遗忘；
  • ②蓝色的是tanh函数，这一部分就与传统的RNN类似了，控制输出范围在-1到1之间有助于管理神经网络。
  • 接下来，将①和②的输出结果相乘，sigmoid激活函数的输出结果会决定是否应该保留tanh激活函数输出的结果


（5）cell state

到目前为止，已经有足够的信息去计算cell state了。首先，cell state会与forget vector逐点相乘。如果相乘的值接近于0则会被遗忘。接下来会把input gate出来的值与刚才的值进行一个相加处理，从而更新cell state到新的值。


（6）output gate

这个output gate会决定下一个hidden state是什么样的。首先，我们把之前的hidden state和现在的输入放入一个sigmoid激活函数，然后会把新的cell state放入一个tanh函数（当然这里是复制一份，见下面的动图）。接下来我们会把tanh的输出结果与sigmoid的输出结果相乘，来决定hidden state应该保存什么内容。此时的输出就是hidden state的状态，走下面的那条管道，用来对下个步骤做出贡献。

五、LSTM的伪代码

总的过程如下：

1.首先，之前的hidden state会与现在的输入串联在一起，将其命名为combine；

2.combine会先被喂入forget layer，这层会移除掉不相关的数据。

3.一个candidate layer会通过combine来创立，这个candidate会保留一个值，用来决定是否通过加法操作加入新的cell state。

4.将combine喂入input layer，这一层会决定candidate中的哪些数据会被加入新的cell state当中。

5.在计算完forget layer，candidate layer和input layer之后，cell state会根据这些向量和之前的cell state来进行计算。

6.计算出output的值。

7.output和新的cell state逐点相乘，可以给出最新的hidden state的状态。

综上，LSTM可以使用隐藏状态进行预测。结合所有这些机制，LSTM可以在序列处理过程中选择要记住或忘记的相关信息。

六、GRU

参考文章：

https://en.wikipedia.org/wiki/Gated_recurrent_unit

GRU是比较新的一代RNN神经网络，结构上比较近似于LSTM。

GRU抛弃了cell state的概念，使用hidden state去传递信息。相对来说，GRU只有两个门控系统：

  • reset gate
  • update gate

结构如下：

（1）Update Gate

与LSTM的forget和input结构类似。他会决定什么信息会被丢弃，什么信息会被加入进来。这样可以用来解决梯度消失的问题。

（2）Reset Gate

这个门用来决定有多少过去的信息要被遗忘掉。

首先，reset gate起作用，它将过去时间步的相关信息存储到新的记忆模块（memory content）中。 然后将输入向量和hidden state与它们的权值相乘。 接下来，它计算reset gate与先前hidden state倍数之间的元素级乘法。 总结以上步骤后，应用非线性激活函数，生成下一个序列。


七、GRU和LSTM

GRU的张量运算较少;因此，他比LSTM训练得更快。不过对于两者来说，目前还没有明确的赢家。研究人员和工程师通常会同时尝试两种方法，以确定哪种方法更适合他们的用例。

  • Manage projects with Repositories

  • Control and track changes with Staging and Committing
  • Branch and Merge to allow for work on different parts and versions of a project
  • Pull the latest version of the project to a local copy
  • Push local updates to the main project

Working with Git

  • Initialize Git on a folder, making it a Repository
  • Git now creates a hidden folder to keep track of changes in that folder
  • When a file is changed, added or deleted, it is considered modified
  • You select the modified files you want to Stage
  • The Staged files are Committed, which prompts Git to store a permanent snapshot of the files
  • Git allows you to see the full history of every commit.
  • You can revert back to any previous commit.
  • Git does not store a separate copy of every file in every commit, but keeps track of changes made in each commit!

一.理论篇

1.MIPS

(1)数据类型

  • 所有的MIPS指令都是32位长
    • 所以半字长=2字节,1字长=4字节,1字节=8位
  • 一个字符空间=1字节
  • 一个整型=1字长=4字节
  • 单个字符用单引号表示,字符串用双引号表示.

(2)寄存器

  • MIPS下共有32个寄存器,标志由$开头.

  • 寄存器表示由两种方式:

    • 第一种:直接使用该寄存器对应的编号,比如$0-$31
    • 第二种:使用对应的寄存器名称,比如$t1,$sp(马上会介绍各个寄存器的名称)

  • 对于乘法和除法分别有两个对应的寄存器,$lo,$hi

    • 对于这两个寄存器,不能直接寻址,必须通过以下指令来分别进行访问对应的内容:
      • mfhi("move from high")
      • mflo("move from low")

注意:栈的走向是从高地址到低地址,而堆的走向是低地址到高地址


(a)MIPS下各个寄存器编号以及描述

Register Number寄存器编号 Alternative Name寄存器名 Description寄存器用途
0 zero 永远返回0
1 $at 汇编保留寄存器(不可做其他用途)
2-3 $v0-$v1 v:value简写,存储表达式或者是函数的返回值
4-7 $a0-$a3 a:argument简写,存储子程序的前4个参数,在子程序调用过程中释放
8-15 $t0-$t7 t:temp简写,函数调用时不保存(但如果需要也可以保存)
16-23 $s0-$s7 s:saved.使用其中一个的子程序必须在退出之前保存原始数据并恢复原始数据。
24-25 $t8-$t9 t:temp简写,算是前面的$t0-$t7的一个继续,属性同$t0-$t7
26-27 $k0-$k1 中断函数返回值,不可做其他用途
28 $gp Global Pointer简写:指向64k($2^{16}$)大小的静态数据块的中间
29 $sp Stack Pointer简写:栈指针,指向的是栈顶
30 $s8/$fp (Saved/Frame Pointer简写)帧指针
31 $ra 用作返回地址

K-means 的算法步骤为：

  1. 选择初始化的 k 个样本作为初始聚类中心

  2. 针对数据集中每个样本$x_i$计算它到 k 个聚类中心的距离并将其分到距离最小的聚类中心所对应的类中

  3. 针对每个类别$a_j$，重新计算它的聚类中心（即属于该类的所有样本的质心）

  4. 重复上面 2 3 两步操作，直到达到某个中止条件（迭代次数、最小误差变化等）。

（3）优缺点分析

  • 优点：
  • 容易理解，聚类效果不错，虽然是局部最优， 但往往局部最优就够了；
  • 处理大数据集的时候，该算法可以保证较好的伸缩性；
  • 当簇近似高斯分布的时候，效果非常不错；
  • 算法复杂度低,比较容易实现。

  • 缺点：
  • K 值需要人为设定，不同 K 值得到的结果不一样；(这一点比较麻烦)
  • 对初始的簇中心敏感，不同选取方式会得到不同结果；
  • 对异常值敏感（也就是对外点敏感）；
  • 样本只能归为一类，不适合多分类任务（比如同时判断出两个特征的分类，如苹果∈红色，苹果∈小体型水果）；
  • 不适合太离散的分类、样本类别不平衡的分类、非凸形状的分类（K聚类更适合于分类出类似圆形/球形的区域，对于不规则形状的处理能力比较差）。

二.网络传输协议

本文中的网络传输协议（Network Transport Protocol），主要是指UDP协议和TCP协议。 在TCP/IP协议族（互联网协议套件，Internet Protocol Suite）中，UDP和TCP位于传输层（Transport Layer），它俩都独立依赖于更底一层网络互连层（Internet Layer）中的IP协议（网际协议，Internet Protocol）。

UDP的特点

从UDP的报文结构当中，我们可以得出其在网络游戏当中使用的特点：

1.不可靠不保序传输（Unsequenced Unreliable Transmission），即传输不保序，也不确保可达，某个具体数据包在传输的过程中，是可能丢失的；也可能数据包到达了，但数据出错从而校验失败，也被Socket丢弃掉。

2.传输快，同样是由于不可靠，导致Socket不会浪费时间和带宽，对丢失的数据包进行重传。

3.连接快，因为根本没有发生连接，所以收发双方每次通信都是无状态的，所以通信前不需建立连接。

4.简单，上层应用能按需实现额外的传输特性（QoS），比如保序不可靠、可靠不保序、可靠保序、最新状态传输（Most Recent State）、快速冗余传输（Quickest Possible Delivery）等（也就是靠应用层来保证UDP的可靠）

5.报头小，节省带宽。

（a）TCP相对UDP额外提供了什么？
  • 保序可靠传输（Sequenced Reliable Transmission），通过报头Sequence number、Acknowledgement Number、Window Size等字段来实现；
  • 流量控制（Flow Control），通过报头的Window Size（窗口大小）实现，接受者把自己的期待流量提示给发送者，防止发送方以过快的速度发送数据给接收方；

（b）TCP在网络游戏中使用的特点

由于TCP的可靠传输，流量控制和拥塞控制技术，其在网络游戏当中可以有如下特点：

（1）本身提供保序可靠传输，不需游戏作额外工作，可以适用于回合制游戏、RPG等通信频率不高的游戏类型；

（2）传输慢，正因TCP本身对所有发出的数据包，都考虑保序可靠、流量控制、拥塞控制（包含了慢开始（Slow-Start），而这些功能对于大部分游戏类型，特别是快节奏（Fast Paced）游戏是不适用的，因为这类游戏通信频率高，要求低通信时延，新发的数据包可能比已发送的数据包更重要（比如游戏状态数据包，游戏状态经常改变，与其重传旧状态数据包，倒不如将其抛弃，赶快传输新的状态数据包）。

（3）连接慢，首次连接需要经过三次握手；

（4）复杂且黑盒，难以进行额外的扩展；

（5）报头比较大，不适用于数据包的量比较大的情况；

综上所述，大多网络游戏采用UDP。 TCP更加适用于回合制等慢节奏游戏，如炉石传说等。 注：有不少MMORPG也使用TCP，尽管如此，未见得TCP是最好的选择。

三.网络同步模型

预知识：什么叫做同步？

比如玩《原神》的时候，要保证组队的人们看到的画面是一样的，以及伤害的数值这些也要是相同的，这就是同步。对于大多数游戏，不仅客户端的表现要一致，而且需要客户端和服务端的数据是一致的。所以，同步是一个网络游戏概念，只有网络游戏才需要同步，而单机游戏是不需要同步的。

最主要的网络模型是：帧同步（Lockstep）和状态同步（State Synchronization）。

对于帧来说，有以下两种术语定义：

  • 逻辑帧：在本文会被称为Tick，游戏在逻辑层面是离散的过程，即可以认为是一个逻辑帧一个逻辑帧地进行逻辑运算，逻辑帧号是指游戏逻辑层面当前处于第几帧；

  • 渲染帧：在本文会被称为Frame，游戏在画面呈现层面也是离散的过程，即可以认为是一幅画面一幅画面地呈现给玩家的，渲染帧号是指游戏当前呈现的是第几幅画面；

其中，游戏的逻辑帧率和渲染帧率是互相独立的，比如一个游戏可以是20帧每秒的逻辑帧率、60帧每秒的渲染帧率。

（1）游戏逻辑的进行

游戏逻辑是一个逻辑帧一个逻辑帧地持续离散进行的，可以抽象为：

  • 游戏在第0个逻辑帧时，根据玩家信息$P$和游戏配置$C$，进行初始化运算$g$，得出初始化状态集合$S_0$

  • 游戏在第$k$个逻辑帧时，根据前一个状态集合$S_{k-1}$和游戏配置$C$，以及第$k$帧收到的外部变化原因集合$I_k$，进行逻辑$t$运算，得出第$k$个逻辑帧新的游戏状态集合$S_k$

其中，对于$I$和$S$的解释如下：

  • $I$:游戏状态变化的根本原因的集合，往往是各个玩家操作；
  • $S$:游戏状态的集合，由众多状态子集组成，其中有以下两个重要子集定义：
    • $O$:一些能被玩家所明显观察到的对象的状态集合;
    • $M$:一些可用于推导最终状态的中间状态集.

（2）状态同步和帧同步概括

在网络同步时，称从客户端发出信息进行网络传输的过程为上行，称客户端经过网络传输收到信息的过程为下行，则，

  • 一般帧同步的本质是，上下行都仅包含游戏外部变化原因集合$I_k$

  • 一般状态同步的本质是，下行仅包含游戏运算得出的结果状态集合$S_k$(确切地说是状态子集$O_k$,也就是能被玩家明显观察到的对象的状态集合)，上行则包含$I_k$和/或状态子集$M_k$;

注：参考的文章当中的帧同步特指只同步操作的确定性锁步同步（Deterministic Lockstep）。不讨论快照同步（Snapshot Synchronization、Snapshot Interpolation），认为其等同于状态同步。

（3）帧同步

由于帧同步只同步游戏外部变化原因集合$I_k$，因此要求各个客户端的运算逻辑$g，t$是严格确定性的，这样所有客户端才能得出一致性的游戏状态结果$S_k$。

  • 如果在计算过程中包含了一丝不确定的因素，即会导致各个客户端运算$S_k$时出现一些误差，那么接下来的逻辑帧误差会越来越大，导致蝴蝶效应，从而最终各个客户端看到的结果状态完全不一样了（比如对浮点数的处理）。

  • 著名帧同步游戏举例：《王者荣耀》

假设玩家本人的客户端和其他正确客户端已经发生不一致，其他玩家在正确客户端作出的合理操作，到达玩家本人的已经状态不一致的客户端做逻辑演算时，却变得不一样的状态结果，这个结果很可能是不合理的，包括数值混乱，操作混乱等。

保证游戏的严格确定性：

根据上述介绍，如果想要保证游戏的严格确定性，可以做出如下的规定：

  • 对浮点数的处理：不使用浮点数而使用定点数，或限定各客户端所运行的硬件及操作系统从而浮点数的运算是一致的。

  • 确定性的随机数机制；

    • 需要保证每次随机的数字都相同，所以需要自己实现一套随机数，不能用unity自带的那个随机数接口，而且需要服务端发送相同的随机种子；因为非常微小的误差就有可能产生蝴蝶效应，所以所有float型的参数必须变成int型，保证计算结果一致。

  • 确定性的容器及算法（增加、移除、排序等）；

  • 隔离和封装逻辑层，以防止其他不确定性的调用；

    • 如需，则也须做到确定性的物理机制、导航机制、动画骨骼机制等，

  • 排查所有引起异常（exception）的逻辑。

制定检测不同步的管线流程的重要性：

对锁步同步游戏来说，不同步造成的游戏体验是极差的，偶现的不同步问题是极为头痛的， 因此制定检测不同步的管线流程对锁步同步游戏来说是至关重要的，比如帧状态哈希对比、静态代码扫描分析、帧级别甚至函数级别的高性能日志、外网不同步率统计，等。

（4）状态同步

因为状态同步只同步游戏运算得出的结果状态$S_k$，所以需要有机器来进行权威（Authoritative）的状态计算，并传输给其他机器，其他机器都将采纳接收到的状态。这种机器就是服务器（或者被称为主机（Host）），而其他没有权威的的机器则被称为客户端。为了简化说明，暂时将服务器集群也抽象为服务器，方便后续的讨论。

状态同步，口头交流中也常不太精确地被称为“CS同步（Client-Server）”。

  • 注：许多著名引擎（如Unreal，Unity）的状态同步的实现都参考了下面这篇文章：Title (gamedevs.org)

状态同步开发过程中最基础也最重要的是，不管客户端网络对象当前处于什么状态，它都要做到能正确地完全退出旧状态，退出后不能残留旧状态的逻辑层效果，并正确地进入服务器告知的新权威状态，从而带来新状态的逻辑层效果。（状态同步的时候上行包含$I_k$和/或状态子集$M_k$）

要避免以非状态同步的方式同步权威状态,比如类似于帧同步，服务器只传输$I_k$给客户端（而不传输$S_k$），让客户端在本地计算出网络对象的新状态$S_k$，这可能会带来服务器客户端之间状态不一致。

比如说，某个网络对象在第$k$个逻辑帧发生了$I_k$而进入到了$S_k$，且之后$S$都不再发生改变，那么服务器只会在第$k$帧才会发送$I_k$；之后若客户端因断线重连或实时死亡重播等原因导致该网络对象状态在客户端被重置为$S_0$，但当前服务器逻辑帧已大于第$k$帧，服务器不会再传输$I_k$给客户端，那么该网络对象在客户端里就错误地一直停留在$S_0$了。

注：之前在帧同步讨论到的确定性指的是“严格的”确定性。在讨论状态同步时，偶尔也会提及“不严格的”确定性（比如最开始作者链接的Q/A阶段最后一个问题），此类确定性只要求客户端服务器之间满足“足够的”确定性，以便客户端能够比较准确地进行预表现即可，因为客户端最终会采纳服务器的状态，修正累计的误差。

（5）帧同步和状态同步的对比

两者的区别

  • 最大的区别就是战斗核心逻辑写在哪:

    • 状态同步的战斗逻辑在服务端;
    • 帧同步的战斗逻辑在客户端。

    战斗逻辑是包括技能逻辑、普攻、属性、伤害、移动、AI、检测、碰撞等等的一系列内容，这常常也被视为游戏开发过程中最难的部分。由于核心逻辑必须知道一个场景中的所有实体情况，所以MMO游戏（例如魔兽世界）就必须把战斗逻辑写在服务端，所以MMO游戏必须是状态同步的，因为MMO游戏的客户端承载有限，并不能把整张地图的实体全部展现出来（例如100米以外的NPC和玩家就不显示了），所以客户端没有足够的信息计算全图的人的所有行为。

  • 具体到客户端和服务端的通信上：

    • 在状态同步下，客户端更像是一个服务端数据的表现层。举个例子，一个英雄的几乎所有属性（例如血量、攻击、防御、攻速、魔法值等等）都是服务端传给客户端的，而且在属性发生改变的时候，服务端需要实时告诉客户端哪些属性改变了，客户端并不能改变这些属性，而是服务端传来什么属性就显示多少属性（虽然可以改变客户端数值达到表现上的效果，例如无限血量，但是服务端那边的血量属性为0时，一样会死）。

      • 此时举一个具体的例子。一个英雄要释放一个非指向性技能（例如伊泽瑞尔的Q），具体过程如下：
        • 客户端通知服务端“我要释放一个技能”——> 服务端通知客户端“在某地以什么方向释放某技能”——>客户端根据这些信息创建一个特效放在某地，然后以某个方向飞行——>服务端根据碰撞检测逻辑判断到某个时刻，这个技能碰到了敌方英雄，通知客户端——>客户端根据服务端信息，删除特效，被打的英雄减血同时播放受击特效。

    • 在帧同步下，通信就比较简单了，服务端只转发操作，不做任何逻辑处理:

      • 现在同一局里有4个玩家，也就是4个客户端，这时客户端A释放了一个技能x，此时将操作传递给服务端，服务端不做任何判断，直接把A的操作全部分发给ABCD，则ABCD同时让客户端A控制的英雄释放技能x。
（a）流量对比

状态同步比帧同步流量消耗大，例如一个复杂游戏的英雄属性可能有100多条，每次改变都要同步一次属性，这个消耗是巨大的，而帧同步不需要同步属性；例如释放一个技能，状态同步下，服务端需要通知客户端很多条消息（必须是分步的，不然功能做不了），而帧同步就只需要转发一次操作就行了。

（b）回放&观战

帧同步的回放&观战比状态同步好做得多，因为只需要保存每局所有人的操作就好了，而状态同步的回放&观战，需要有一个回放&观战服务器，当一局战斗打响，战斗服务器在给客户端发送消息的同时，还需要把这些消息发给回放&观战服务器，回放&观战服务器做储存，如果有其他客户端请求回放或者观战，则回放&观战服务器把储存起来的消息按时间发给客户端。

（c）安全性

状态同步的安全性比帧同步高很多，因为状态同步的所有逻辑和数值都是在服务端的，如果想作弊，就必须攻击服务器，而攻击服务器的难度比更改自己客户端数据的难度高得多，而且更容易被追踪，被追踪到了还会有极高的法律风险。而帧同步因为所有数据全部在客户端，所以解析客户端的数据之后，就可以轻松达到自己想要的效果，例如moba类游戏的全图挂，某些吃鸡游戏的透视挂，都是没办法防止的，不过更改数据达到胜利的作弊方式（例如更改自己的英雄攻击力）可以通过服务器比对同局其他人的战斗结果来预防。

（d）服务器压力

状态同步服务器压力比较大，因为要做更多运算。

（e）开发效率

首先要说，状态同步的游戏占主流，其次就是状态同步开发起来比较难。

  • 帧同步服务器开发难度低，同一套方案可以给很多不同类型的游戏使用，反正都是转发操作；减少了服务端客户端沟通，老实说，没有扯皮的时间，开发效率最起码提高20%。
  • 状态同步的方案下，同一个功能至少需要一个客户端和服务端共同完成；PVP和PVE基本用的是同一套代码，做完PVP很容易就可以做单机的PVE。
（f）断线重连
  • 状态同步的断线重连很好做，无非就是把整个场景和人物全部重新生成一遍，各种数值根据服务端提供加到人物身上而已。
  • 帧同步的断线重连就比较麻烦了，例如客户端在战场开始的第10秒短线了，第15秒连回来了，就需要服务端把第10秒到第15秒之间5秒内的所有消息一次性发给客户端，然后客户端加速整个游戏的核心逻辑运行速度（例如加速成10倍），直到追上现有进度。
